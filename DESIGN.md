## 0. Headline decision: lane diverges from the catalog (oauth_light → api_key)

The catalog lists Mastodon as `oauth_light`. The 2026-07-21 OAuth audit does
**not** cover it (the audit only re-laned tools that sat in `api_key`; Mastodon
was assigned oauth_light directly). Independent verification against the
official Mastodon docs shows `standard_oauth` is a poor fit, and the master plan
already flags exactly this: *"Mastodon … is federated, and OAuth apps register
per-instance (dynamic client registration), so lane 1's one-app-one-client-secret
model does not fit — either scope the provider to a single flagship instance
(e.g. mastodon.social) or build a dynamic-registration adapter"* (§6). Verified
facts driving the divergence:

1. **Every Mastodon server is its own OAuth authorization server.** There is no
   central Mastodon-operated auth server. The official spec is explicit: *"allow
   your users to specify the domain they want to connect to before login. Use
   that domain to acquire a client id/secret."*
   (<https://docs.joinmastodon.org/spec/oauth/>) The bundle's static
   `oauth.authorize_url` / `token_url` are single fixed strings — they cannot be
   "the user's instance."
2. **Client credentials are per-instance and must be minted on each instance.**
   You obtain `client_id`/`client_secret` by calling `POST /api/v1/apps` **on
   that specific server** (anonymous, no review). Mastodon *"does not support the
   Dynamic Client Registration protocol, however it does support a proprietary
   endpoint for registering an OAuth Application"* — so there is no way to
   pre-register **one** Helio client that arbitrary instances honor.
   integration-service config carries exactly one `oauth.client_id` /
   `client_secret` per provider (lane 1's model); it has no slot for a
   per-connection, dynamically-minted pair.
3. **Confidential clients only; headless bots are not the recommended shape.**
   Mastodon *"presently supports only confidential clients"*; PKCE exists (4.3.0+)
   but the model is a redirect flow tied to a server the client already has
   credentials on. An AI teammate posting on a user's behalf is headless
   automation across an open federation.

This is the **audit rubric's own escape clause** — *"OAuth that is per-instance …
or otherwise impractical for a shared client keeps the tool in `api_key`."*
Mastodon OAuth is per-instance by definition.

**Official-doc-supported alternative — the self-serve Application access token
(the standard bot path).** Mastodon ships a first-class credential for exactly
this use case. On the user's own instance, **Settings → Development → New
Application** (`https://<instance>/settings/applications/new`): name it, pick
scopes (`read`/`write` or granular `read:statuses`/`write:statuses`/`write:media`/
`write:favourites`), submit, then open the application and copy **"Your access
token."** No redirect flow, no review, works on **any** instance, generated by
the account owner in ~30 seconds
(<https://docs.joinmastodon.org/client/token/>, corroborated by the widely-used
bot setup guides). The token is a plain `Authorization: Bearer` credential and,
per Mastodon's app-token model, **does not expire and has no refresh token** —
it is revoked only when the user deletes the application. This maps cleanly onto
Helio's existing `manual_credentials` + api_key infrastructure (the
`mongodb`/`servicenow`/`bluesky` precedent: a static secret the user pastes,
stored in Vault, served verbatim by the token gateway).

Because the token is **instance-scoped**, the credential is **two inputs** — the
**instance URL** (part of every request URL) **plus** the **access token** —
exactly the ServiceNow "endpoint + secret" shape (§0a, §4).

**Recommendation:** ship Mastodon as **api_key lane, `manual_credentials`
(endpoint + secret)** — fields `instance_url` + `access_token`. This is the
**Bluesky-class auth-shape divergence** (the sibling federated-social tool in
this program resolved oauth_light → api_key/app-password for the same "per-server
OAuth impractical" reason). Record it in the §6 amendment log. Row, id, key,
wave (3-hold), and the 298-total invariant are unchanged; only the auth **type**
moves oauth_light → api_key. **Lane-1 work for this tool drops to zero** (no
OAuth app registration on any instance, no review clock) — a net simplification
for the holdback batch.

**Rejected — Option A, flagship-instance-only OAuth (`standard_oauth` against
mastodon.social).** Register one app on mastodon.social, hardcode its
authorize/token URLs + client id/secret. Fits `standard_oauth` with zero
adapter, but **strands the entire federation**: only mastodon.social accounts
could ever connect, defeating the point of a federated network and surprising
users on fosstodon/hachyderm/self-hosted servers. The whole value of Mastodon is
that it is *not* one instance.

**Rejected — Option B, dynamic-registration OAuth adapter.** At connect time,
POST `/api/v1/apps` to the user's instance to mint per-connection client
credentials, then drive authorize/token against per-instance URLs. This needs:
per-connection client id/secret storage (not one config value), per-connection
authorize/token URLs (not static bundle strings), and a DCR step before
authorize — materially outside `standard_oauth`, a multi-week infra project
(the master plan's "dynamic-registration adapter" / NetSuite-class flag). The
access-token path is the documented, lower-friction, federation-native bot
integration and is strictly better for our use case.

### 0a. Implementation divergence: endpoint+secret manual credentials verified NOT on the worktree base (2026-07-22)

The two-input shape (`instance_url` non-secret + `access_token` secret) requires
the **"endpoint + secret" `manual_credentials`** capability. Verified against the
integration-service on this worktree's fork point: **it is single-secret only.**

- `model/runtime_contract.go:328` hard-fails `len(input.Fields) != 1 ||
  !input.Fields[0].Required` ("must be exactly one required field, design 317
  D5").
- `service/manual_credential.go:179` re-asserts the single-field rule; storage is
  one secret through `writeUserTokenCredential`; the only token source is
  `token.access_token`.
- `resolveManualSecret(definition, rawToken, values map[string]string)` already
  receives the full `values` map keyed by field name — the plumbing hook for a
  second field exists, but the contract rejects it today.

**This is the identical gap the `servicenow` branch designs and closes** (its
§4.3: relax D5 to "one secret + one required non-secret endpoint field"; carry
the endpoint through the manual-credential Connect path; add an instance-identity
deriver; relax the L4 seed gate to `AuthCredentials`). ServiceNow explicitly
scopes it as a **reusable OQ3 "self-hosted URL + token" capability, not a
servicenow special-case** — Mastodon (federated "instance URL + token") is the
**same class and its natural second consumer.**

**Capability-reuse decision for the batch lead (verify against `main` at build
time — reuse before adding):**

1. **Preferred — reuse the ServiceNow endpoint+secret capability** if it has
   landed on `main`. Mastodon's bundle then needs **zero** new integration-service
   code beyond a `mastodonAccountVerifier` deriver (calls `verify_credentials`,
   §3) — and even that may be a parameterization of ServiceNow's
   `instanceIdentityDeriver` (same "verify against `<endpoint>` with the secret,
   derive account_key + label" contract). Confirm whether multi-field manual
   credentials (the `zoominfo`/`mixpanel` two-field storage, per completed
   program tasks) also reached `main`; if so, storing `instance_url` +
   `access_token` as two projected fields (account_key = full account URL,
   §3.4) is even cleaner than endpoint+secret.
2. **Fallback — combined single secret** (the `bluesky` §0a precedent) if neither
   capability is on `main` at build time: store `instance_url` and `access_token`
   as one secret joined by a space (`https://mastodon.social AT...token`) via the
   existing `token.access_token` source; split on the first space in two places —
   the anycli service reads one env var and splits it, and a
   `mastodonAccountVerifier` splits the same secret at connect time. A URL never
   contains a space and Mastodon access tokens are URL-safe base64 (no spaces), so
   the first-space split is unambiguous. This ships end-to-end on the base with
   zero gateway/enum growth; collapsing 2 fields → 1 is the only concession, and
   splitting them back is a follow-up once the capability lands.

Either way lane/row/id/key, the `verify_credentials` verification, and the
five-layer plan are preserved. The rest of this doc is written for the
**endpoint+secret** shape (path 1) and notes the fallback inline.

**Build-time resolution (2026-07-22, shipped): path 2 — combined single
secret.** Re-verified the worktree base (`main`): `validateCredentialInputSchema`
(runtime_contract.go:328) and `resolveManualSecret` (manual_credential.go:179)
still enforce **exactly one required field**, and neither the ServiceNow
endpoint+secret relaxation nor the zoominfo/mixpanel multi-field storage is on
`main` (their provider bundles are absent). Building either shared capability in
the Mastodon branch would fork a surface another branch owns, so Mastodon ships
the **combined-secret fallback**, self-contained on the base: one required
secret field `access_token` carrying `"<instance-url> <token>"`, projected via
the existing `token.access_token` source, split on the first space in **both**
places — the AnyCLI service (`splitCredential`, reads one env var) and a new
`mastodonAccountVerifier` (splits, then `verify_credentials`, derives
account_key = normalized instance URL + `@user@host` label). The verifier is
selected by provider key in the manual_credentials branch of
`composeProviderRegistration` (keyed off a string-literal `model.Provider`
constant so integration-service compiles before the batch-end regen). No
gateway/enum growth; provider-gen projections generated + `--check`-clean
locally but **not committed** (rollout plan §2). Follow-up once the
endpoint+secret capability lands: split `access_token` back into two fields
(`instance_url` non-secret + `access_token` secret) — a UI-only change; the
verifier and account_key contract are unchanged.

## 1. Official API surface wrapped, and why

Mastodon's REST API is a single, uniform surface hosted **on each instance**
(`https://<instance>/api/v1/...`, plus a few `/api/v2/...`). Auth is a bearer
token on every call. What an AI teammate actually does with Mastodon — post,
read the timelines, search, look up people, engage, and check mentions — drives
the endpoint selection (all verified against docs.joinmastodon.org):

| Capability | Method + endpoint | Scope |
|---|---|---|
| Verify / identity | `GET /api/v1/accounts/verify_credentials` | `read:accounts` |
| Post a status (text, reply, CW, visibility, poll) | `POST /api/v1/statuses` | `write:statuses` |
| Post with media | `POST /api/v2/media` (upload) → `POST /api/v1/statuses` (attach ids) | `write:media` + `write:statuses` |
| Delete a status | `DELETE /api/v1/statuses/:id` | `write:statuses` |
| Read a status / context (thread) | `GET /api/v1/statuses/:id`, `GET /api/v1/statuses/:id/context` | `read:statuses` |
| Home timeline | `GET /api/v1/timelines/home` | `read:statuses` |
| Public / local timeline | `GET /api/v1/timelines/public` | `read:statuses` |
| Hashtag timeline | `GET /api/v1/timelines/tag/:hashtag` | `read:statuses` |
| Search (accounts, hashtags, statuses) | `GET /api/v2/search?q=…&type=…` | `read:search` |
| Account lookup / by handle | `GET /api/v1/accounts/lookup?acct=…`, `GET /api/v1/accounts/:id` | `read:accounts` |
| A user's posts | `GET /api/v1/accounts/:id/statuses` | `read:statuses` |
| Favourite / boost | `POST /api/v1/statuses/:id/favourite`, `/reblog` | `write:favourites`, `write:statuses` |
| Follow / unfollow | `POST /api/v1/accounts/:id/follow`, `/unfollow` | `write:follows` |
| Notifications (mentions, follows, boosts) | `GET /api/v1/notifications` | `read:notifications` |

**Recommended token scopes** for the setup doc: request **`read` + `write` +
`follow`** (the three broad categories cover every verb above without the user
hand-picking sub-scopes). Users who prefer least-privilege can select the
granular sub-scopes in the table.

**Provider-specific detail the service handles so the AI doesn't:**

- **Idempotency.** `POST /api/v1/statuses` honors an `Idempotency-Key` header —
  the service sends one per post-create so a retried invocation never
  double-posts. This matters because each `heliox tool` call is a fresh process.
- **Pagination is Link-header based**, not offset — Mastodon returns
  `max_id`/`min_id` cursors in the `Link` response header. The service parses it
  and surfaces a `next`/`prev` cursor in JSON so the AI paginates without reading
  raw headers.
- **Media is a two-step, sometimes-async upload.** `POST /api/v2/media` may return
  `202 Accepted` while processing; the service polls `GET /api/v1/media/:id`
  until ready before attaching to a status (bounded), and passes `--alt` through
  as `description` (accessibility; default empty with a warning). Cap at 4
  attachments (provider limit).

Status `id`s are opaque per-instance strings the AI passes back verbatim for
delete/reply/favourite — the service echoes them in output. **Not wrapped in v1**
(reachable later): lists, bookmarks, filters, scheduled statuses, admin API — all
lower-frequency for a teammate; a raw `api` escape hatch (below) keeps them
reachable.

## 2. anycli definition (axis ② id: `mastodon`)

**Type: `service`** (stage-1 rubric). No official Mastodon CLI binary exists that
is non-interactive, `--json`-capable, and provisionable into the runtime image
(`toot`/`madonctl` are third-party, interactive, and not first-party); the
integration is HTTP against the REST API. Lives in `internal/tools/mastodon/`
(package `mastodon` — id has no dashes and no leading digit, so package name ==
id verbatim), registered `RegisterService("mastodon", &mastodon.Service{})` in
`internal/tools/register.go`. Copy the `notion`/`servicenow` service shape:
`BaseURL`/`HC`/`Out`/`Err` struct for `httptest` injection, cobra tree grouped by
resource, exit-code contract (0 success, 1 runtime/API failure via typed
`apiError`, 2 usage/parse), `--json` structured error envelope.

**Base URL from credential (mongodb/servicenow precedent).** Unlike a constant-
host service, Mastodon's target host is per-connection — it comes from the
injected `instance_url`. The service builds `<instance_url>/api/v1/...` at request
time; tests point `BaseURL` at an `httptest.Server`.

**Definition `definitions/tools/mastodon.json`** — two credential fields injected
as env; the service reads them and calls the instance directly:

```json
{
  "name": "mastodon",
  "type": "service",
  "description": "Mastodon as a tool (post, read timelines, search, engage; per-instance access token)",
  "auth": {
    "credentials": [
      { "source": { "field": "instance_url" },
        "inject": { "type": "env", "env_var": "MASTODON_INSTANCE_URL" } },
      { "source": { "field": "access_token" },
        "inject": { "type": "env", "env_var": "MASTODON_ACCESS_TOKEN" } }
    ]
  }
}
```

anycli has **no** single-secret restriction — a definition may bind multiple
credential fields (the `servicenow` two-field precedent). The service reads
`MASTODON_INSTANCE_URL` (→ derived base URL) and `MASTODON_ACCESS_TOKEN` (→
`Authorization: Bearer` header). Field names `instance_url` / `access_token` are
the contract the Helio bundle projects into (§4). *(Combined-secret fallback,
§0a path 2: one field `access_token` carrying `"<instance_url> <token>"`, one env
var, split in the service.)*

**Command tree (verbs).** Grouped by resource, `--json` on every leaf:

```
mastodon whoami                                            # verify_credentials
mastodon post create --text "…" [--reply-to <id>] [--cw "…"]
                     [--visibility public|unlisted|private|direct]
                     [--image PATH --alt "…"]… [--lang en]
mastodon post delete --id <id>
mastodon post get    --id <id>                             # status + context (thread)
mastodon timeline home           [--limit N] [--cursor C]
mastodon timeline public [--local] [--limit N] [--cursor C]
mastodon timeline tag <hashtag>  [--limit N] [--cursor C]
mastodon account get    <acct|id>                          # lookup by @handle or id
mastodon account posts  <acct|id> [--limit N] [--cursor C]
mastodon search --q "…" [--type accounts|hashtags|statuses] [--limit N]
mastodon favourite --id <id>
mastodon boost     --id <id>
mastodon follow    <acct|id>
mastodon unfollow  <acct|id>
mastodon notifications list [--limit N] [--cursor C]
mastodon api <METHOD> <path> [--body '<json>'] [--query k=v]…   # raw escape hatch
```

- `account get/posts/follow` accept a human `@user@instance` handle and resolve
  it to a numeric account id via `GET /api/v1/accounts/lookup?acct=…` — the AI and
  humans speak handles, not per-instance numeric ids.
- `api` mirrors notion/servicenow's raw verb (any `/api/...` path, method, body,
  query) and **rejects** attempts to override the `Authorization` header
  (notion's "cannot be overridden" precedent).

**JSON output shape.** Provider-neutral, thin, agent-consumable — never raw
Mastodon passthrough. Examples:

- `post create` →
  `{"id":"111…","url":"https://mastodon.social/@alice/111…","visibility":"public","created_at":"…"}`
- `timeline` / `search statuses` →
  `{"posts":[{"id","url","account":{"id","acct","display_name"},"content_text","created_at","replies_count","reblogs_count","favourites_count"}],"cursor":"111…"}`
  (`content_text` = HTML stripped to plain text so the AI reads posts without
  parsing markup).
- `account get` →
  `{"id","acct","display_name","note_text","followers_count","following_count","statuses_count","url"}`
- error →
  `{"error":{"code":"api_error","message":"…","status":422,"provider_error":"Validation failed: …"}}`
  (exit 1). Mastodon returns `{"error":"…"}` (and sometimes
  `{"error":"…","error_description":"…"}`); the service maps both.

**TDD (L1).** `httptest` fake serving `verify_credentials` + each endpoint;
assert request method/path/query, injected `Authorization: Bearer` header,
derived base URL from `instance_url`, `Idempotency-Key` on post-create, Link-header
cursor parsing, media upload→`202` poll→attach wiring, handle→id lookup, HTML→text
stripping, `api` auth-override rejection, and both plain-text and `--json` error
rendering + exit codes. Never hit the live API in a unit test.

## 3. Credential fields & exact auth flow (api_key lane, verified)

**Credential the user supplies:** two fields.

| Field | Label | Secret | Notes |
|---|---|---|---|
| `instance_url` | Instance URL | no | e.g. `https://mastodon.social` — the user's home server |
| `access_token` | Access token | yes | From Settings → Development → your application → "Your access token" |

`setup_url` → the new-application page **relative to the user's instance** cannot
be a single static URL (federation); point at the canonical docs
(`https://docs.joinmastodon.org/client/token/`) and state the pattern
`https://<your-instance>/settings/applications/new` in the field help + AI-facing
doc. Recommend scopes **read, write, follow**.

**Auth flow at connect time.** No OAuth redirect. The user pastes instance URL +
access token into the connect drawer → `POST /connections/credentials`
(write-only, stored in Vault). Identity + verification is a **session/account
verifier** (`mastodonAccountVerifier`, the endpoint+secret deriver from §0a/§4):
integration-service calls
`GET <instance_url>/api/v1/accounts/verify_credentials` with
`Authorization: Bearer <access_token>`; on `200` it reads `acct`/`username` +
`url` (human label + stable key) and `id`; on `401` it rejects the credential
(`invalid_provider_credential`). This is strictly better than mongodb's no-verify
stance because Mastodon *has* a concrete per-instance identity endpoint — the
pattern mirrors `servicenow`/`crisp` verifiers that call a provider endpoint to
derive identity.

**Auth flow at runtime.** Token gateway serves the stored `{instance_url,
access_token}` verbatim as the provider-neutral credential map; anycli injects
them as env; the service calls the instance with `Authorization: Bearer`. **No
refresh cycle** — the app token is a long-lived static secret (revoked only when
the user deletes the application in Mastodon settings). Seed `access_token` only,
omit `refresh_token`/`expires_at` (the Slack-non-expiring-token class in the
skill's "picking a token per provider class").

**Account key / identity (§3.4).**

- **Endpoint+secret path (ServiceNow shape):** `account_key` = the **normalized
  instance base URL** (`https://<instance>`, scheme+host, no trailing slash),
  which also doubles as the value anycli needs for the base URL; the verifier
  sets the human **label** to `@username@instance` from `verify_credentials` so
  the UI shows the account, not just the host. v1 limitation: one Mastodon
  connection per **instance** per assistant (two accounts on the same server
  collapse to one connection) — acceptable for a teammate that connects one
  identity; different instances are naturally distinct connections. Documented as
  a follow-up.
- **Multi-field path (if zoominfo-style two-field storage is on `main`):**
  `account_key` = the account's full `url`
  (`https://mastodon.social/@alice`, globally unique per account), with
  `instance_url` stored as its own projected field — gives true per-account
  uniqueness. Prefer this if the capability is available.

## 4. Helio provider bundle plan (`integrations/providers/mastodon/provider.yaml`)

**Three naming axes — all identical, zero divergence**, so **no `toolToProvider`
entry** is added to `helio-cli/internal/toolcred/resolver.go`:

| Axis | Value | Where |
|---|---|---|
| ① CLI command word | `mastodon` (flat, ungrouped) | bundle `tool.command` (defaults to name) |
| ② anycli tool id | `mastodon` | `definitions/tools/mastodon.json` |
| ③ provider catalog key | `mastodon` | bundle dir `integrations/providers/mastodon/` |

Go package `mastodon` (no dash normalization needed).

**Bundle shape** (manual_credentials endpoint+secret, hidden-first), modeled on
`servicenow`:

```yaml
schema: helio.provider/v1
key: mastodon
go_name: Mastodon

presentation:
  name: Mastodon
  description_key: mastodon
  consent_domain: joinmastodon.org
  visible: false            # hidden-first; flip is the single go-live change

auth:
  type: credentials
  owner: individual
  credential_input:
    fields:
      - name: instance_url
        label_key: mastodon_instance_url
        secret: false
        placeholder: "https://mastodon.social"
        required: true
      - name: access_token
        label_key: mastodon_access_token
        secret: true
        placeholder: "Your access token (Settings → Development)"
        required: true
    setup_url: https://docs.joinmastodon.org/client/token/

identity:
  source: strategy          # mastodonAccountVerifier: verify_credentials → account_key + label

connection:
  mode: isolated
  disconnect_mode: local_only   # Mastodon has no token-revoke API for app tokens; user deletes the app
  runtime_strategy: manual_credentials

resources: { selection: none, discovery: none, enforcement: none }

credential:
  fields:
    access_token: token.access_token      # the secret
    instance_url: connection.account_key   # the instance base URL (== account_key, endpoint+secret path)
tool:
  name: mastodon
  kind: api-key             # wire-compat value; auth_type routes the drawer
```

`credential.fields` uses **only existing** closed-vocabulary sources
(`token.access_token`, `connection.account_key`) — **no new `CredentialSource`**.
The token gateway hands anycli `{access_token, instance_url}`, matching §2.
*(Multi-field path: add an `instance_url` projected field + `account_key` = full
account url. Combined-secret fallback: single `access_token` field carrying
`"<instance_url> <token>"`.)*

**Capability growth (integration-service).** Depends on the **endpoint+secret
`manual_credentials`** capability (§0a) — verified **not** on the worktree base;
**reuse the `servicenow` capability from `main` if landed**, else the batch lead
lands it once and Mastodon + ServiceNow (+ OQ3 Jenkins/Metabase/Rocket.Chat/
Mattermost) share it. The only tool-specific piece is `mastodonAccountVerifier`
(calls `verify_credentials`, extracts label + account_key) — first check whether
ServiceNow's `instanceIdentityDeriver` can be parameterized instead of adding a
new deriver (grow a reviewed enum value over a new adapter). Also confirm the L4
**seed gate** accepts `AuthCredentials` (ServiceNow §4.3 item 4) — required for
L4 seeding.

**No integration-service OAuth config, no lane-1 app registration, no `deploy/`
Secret append** — the api_key lane needs none. Config Sync hard rule is trivially
satisfied (nothing to sync). This removes Mastodon from lane 1's 3-hold
registration queue entirely.

**Other hidden-first artifacts** (batch-end merge): `internal/tools/register.go`
entry, UI icon `ui/helio-app/src/integrations/icons/mastodon.svg` +
`providerIcons.ts` append, i18n `tools.desc.mastodon` + the two `label_key`
strings (`mastodon_instance_url`, `mastodon_access_token`) across **all 9 locales**
(`en-US`, `zh-CN`, `zh-TW`, `de-DE`, `es-ES`, `fr-FR`, `ja-JP`, `ko-KR`, `pt-BR`),
AI-facing sub-doc under `agents/plugins/heliox/skills/tool/`, one plugin version
bump.

## 5. Test plan — five layers
