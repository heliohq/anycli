# Expensify tool — DESIGN

**Catalog row:** #146 Expensify · anycli id `expensify` · provider key `expensify` ·
auth lane `api_key` · wave 2 · category Finance
(`docs/design/008-300-integrations-rollout-plan.md` §4).

**Audit verdict:** `oauth-audit.md` row 148 — *"no viable multi-tenant path … Stays
api_key per rubric."* Confirmed against the official docs below: Expensify's only
programmatic auth is a per-account `partnerUserID` / `partnerUserSecret` pair the
user generates for themselves; there is no authorization-code OAuth a single Helio
app could offer to arbitrary Expensify accounts. **Verdict upheld — `api_key`.**

## 1. Official API (verified 2026-07)

Source: <https://integrations.expensify.com/Integration-Server/doc/> (Expensify
Integration Server API).

- **Single endpoint.** `POST https://integrations.expensify.com/Integration-Server/ExpensifyIntegrations`.
  Every job — read or write — hits this one URL. There is **no** REST resource
  tree and **no** GET identity endpoint.
- **Request encoding.** `application/x-www-form-urlencoded` (or multipart) with a
  single form field `requestJobDescription` whose value is a JSON document:
  ```json
  {
    "type": "get",
    "credentials": { "partnerUserID": "...", "partnerUserSecret": "..." },
    "inputSettings": { "type": "policyList", "adminOnly": true }
  }
  ```
  Top-level `type` is one of `get | create | update | file | download | reconciliation`.
  `credentials` is identical across every job.
- **Credentials.** A `partnerUserID` + `partnerUserSecret` **pair**, generated by
  the user at <https://www.expensify.com/tools/integrations/> ("shown once, store
  securely"). Both are required on every call. Neither is an OAuth token; there is
  no refresh and no expiry lifecycle.
- **Errors.** Responses are HTTP 200 with a JSON body carrying `responseCode`
  (e.g. `200` OK, `401` bad credentials, `410`/`4xx` other). Rate limit: 5 req /
  10 s and 20 req / 60 s → HTTP `429`.
- **Concrete jobs used by the initial surface:**
  - Policy List getter — `type:get`, `inputSettings.type:policyList` (optional
    `adminOnly`, `userEmail`).
  - Policy getter — `type:get`, `inputSettings.type:policy`, `policyIDList` (array,
    required), optional `fields` (`categories|reportFields|tags|tax|employees`),
    `userEmail`.
  - Everything else (report export `file` → `download`, `create`, `update`,
    `reconciliation`) is reachable through a raw escape-hatch command (§3).

## 2. Credential model — colon-pair single secret (the key decision)

Expensify needs **two** values at call time. Helio's manual-credential write path
on this worktree's base is **single-secret** by design (design 317 D5: the
generator, `model.validateCredentialInputSchema`, and `resolveManualSecret` all
assert *exactly one required field*; the vault stores one `access_token`).

Two shapes were available for a credential pair:

- **A — grow multi-field manual_credentials storage** (relax D5, store N secrets,
  project N credential sources). This is a large cross-cutting change (model,
  generator, write path, vault, token gateway).
- **B — combine the pair into one opaque secret** `partnerUserID:partnerUserSecret`,
  keep single-secret storage, split it where it is consumed.

**Decision: B.** Rationale:
- It stays inside the base's shipped single-secret capability — no change to the
  D5 storage boundary, the generator contract, the vault schema, or the token
  gateway.
- It is already an established batch pattern for credential pairs: crisp
  (`keypair identity deriver`) and amplitude (`first-colon-split identity
  deriver`) store an `id:secret` pair as one secret and split it. Expensify joins
  that precedent rather than reinventing multi-field storage (which other tools —
  zoominfo/paypal/mixpanel/braintree — grow canonically elsewhere and which the
  batch lead reconciles; a second independent multi-field implementation here
  would only add merge surface).
- The two values are colon-safe: Expensify's `partnerUserID` is a bare token /
  email and `partnerUserSecret` is alphanumeric — neither contains a colon — so
  splitting on the **first** colon is total and lossless.

### Divergence recorded
This is the smaller, orthogonal choice, **not** the batch's multi-field-form
choice for pairs. It is documented here per the master-plan requirement to record
auth-shape divergences. If/when the canonical multi-field manual_credentials
capability lands on `main`, Expensify *may* be migrated to two discrete input
fields before its visible flip (a hidden-only, pre-GA cosmetic change; the account
key stays `partnerUserID` either way). No stored-data migration is owed while the
tool is hidden and unshipped.

### Where the split happens
- **Helio (identity only).** A new `keyPairPrefixIdentityDeriver` (mirrors the
  existing `dsnHostIdentityDeriver`: no provider-side call — Expensify has no GET
  identity endpoint) derives the stable account key + human-readable label from
  the secret prefix before the first colon (= `partnerUserID`). A bad pair
  surfaces at first use via AnyCLI's `CredentialRejected`, exactly like the
  mongodb DSN precedent (design 317 OQ1 no-verify / OQ2 readable key).
- **anycli (execution).** The token gateway injects the whole
  `partnerUserID:partnerUserSecret` as one env var `EXPENSIFY_CREDENTIALS`; the
  service splits on the first colon and builds the `credentials` object. AnyCLI is
  the only place that needs both halves at once.

## 3. anycli command surface (`type: service`)

Non-interactive cobra tree, `--json` everywhere, design-318 side-effect
annotations on every leaf:

- `expensify policy list [--admin-only] [--user-email <e>]`
  → `{type:get, inputSettings:{type:policyList,…}}` (read, `side_effect:false`).
- `expensify policy get --policy-id <id>… [--field <f>…] [--user-email <e>]`
  → `{type:get, inputSettings:{type:policy, policyIDList:[…], fields:[…]}}`
  (read, `side_effect:false`).
- `expensify request --input <json> [--top-type get|create|update|file|download|reconciliation]`
  → the escape hatch: the caller supplies the `requestJobDescription` body
  **without** `credentials`; the service injects credentials and POSTs it
  verbatim. Covers report export/download, create, update, reconciliation without
  bespoke flags. Marked `side_effect:true` (it can mutate).

The service never accepts credentials as flags/args (AGENTS.md no-interactive /
no-secrets-in-argv): they arrive only through `EXPENSIFY_CREDENTIALS`. Missing or
malformed (`no colon`) env → exit 1 with guidance. Response bodies are emitted
verbatim; `responseCode` 401/403 → `execution.RejectCredential`, other `>=400`
→ error carrying the provider message.

## 4. Naming & wiring

- Axis ① CLI word `expensify` · ② anycli id `expensify` · ③ provider key
  `expensify` — **identical**, so **no** `toolToProvider` resolver entry (only
  ②↔③ divergences get one).
- Go package `internal/tools/expensify` (id has no dashes/leading digit).
- Registered in `internal/tools/register.go` as `RegisterService("expensify", …)`.

## 5. Helio bundle (hidden-first)

`integrations/providers/expensify/provider.yaml`: `auth.type: credentials`, one
`credential_input.fields` entry `credentials` (secret, placeholder
`partnerUserID:partnerUserSecret`, setup_url = the integrations tools page),
`identity.source: strategy`, `connection.runtime_strategy: manual_credentials`,
`connection.disconnect_mode: local_only`, `credential.fields:
{credentials: token.access_token, account_key: connection.account_key}`,
`presentation.visible: false`. Ships hidden until the pin bump, five-layer test
pass, and (only then) the visible flip.

  mapping).

- **L3** `provider-gen --check` + both repos' unit suites (run locally; projections
  not committed).
- **L4** singleton + seeded credential (`partnerUserID:partnerUserSecret`) through
  the real token gateway.
- **L5** api_key key-entry connect flow before the visible flip.
